DBC Week 8: What is Rails?

And the topic is: Rails!

For this week's topic, we had a multitude of options to research and write on. Not surprisingly, I chose Rails, since it appears to be where all the "buzz" is, kind of like Steph Curry in basketball right now.

Rails is a framework for web app development

Follow up question, what's a framework?

Wikipedia uses some fancy words for this like "abstraction" that don't make a ton of sense for somebody who speaks regular English, so we'll try some other words. In short, a software framework is a standardized way of thinking when it comes to building "something."

The "something" in this case is a web application (different from a webpage). The people who developed Rails had very specific ideas on how web applications should be written, so they developed Rails so that others could follow in their footsteps.

And finally, the reason we call it Ruby on Rails, is because we need to write Ruby to utilize the power of Rails.

But what does it do?

In the fewest words possible, it makes building web apps faster.

The Rails documentation talks a lot about Rails being opinionated. That is, the folks who developed Rails have a specific idea of what best practice means when it comes to developing a web application, and have created shortcuts to help make building web apps very quick and efficient.

An example of this might be that they assume the web app will need to interact with a SQL database, and interact with the SQL database in a specific, and prescriptive way. As a result, they've pre-written a bunch of code that makes it easier for our web-app to interact with a database, with the key caveat being that we are going to have to interact with the database in the way that they want us to.

So how does this make things that we see on the internet

So far, we've learned that there are 3 major components of a webpage (major paraphrasing here):

  • HTML: Puts words and pictures on a page
  • CSS: Dresses up those words, puts them in boxes, and tells the boxes where to be on a page
  • JavaScript: Helps animate, change words, and move boxes around dynamically

And here's where Rails comes in:

  1. You go to a web page (www.something.com)
  2. www.something.com sends a request to a server (far far away) asking for what it needs to show you.
  3. The server uses Rails to compile HTML / CSS / JS snippets into full web pages that it send back to your browser (the client) for you to view

That sounds kind of inefficient!

Maybe it is, but the key here is that web applications are VERY complex. There are lot of moving pieces that are tough to handle on the client side (your browser). Think about any web app and how much they need to think about - from login and user information, dealing with live social media streams, to merchant engines for buying things, maybe a chat functionality...and so on.

Furthermore, browsers are a bit stupid (scathing!) in that they can only read HTML, CSS, and JS, likely so that they don't crash trying to do too much.

It actually becomes faster to route all of the heavy lifting to a remote server to think about what pages to show you the user.

Ruby does the lifting...

...and Rails are the steroids.

Conclusion

The takeaway here is that Rails has become immensely popular because of how quickly developers can take concept and put it on the internet with a wide range of features. Because it's opinionated, there are streamlined "best practice" ways to do things that don't require building a lot of extra support functionality because it's been done for you! I'm very excited to start learning Rails, to say the least.
